## <a name="what-aspnet-core-module-does"></a><span data-ttu-id="52b51-110">ASP.NET Core 模組的功能</span><span class="sxs-lookup"><span data-stu-id="52b51-110">What ASP.NET Core Module does</span></span>

<span data-ttu-id="52b51-111">ANCM 是連結到 IIS 管線，並將流量重新導向至後端 ASP.NET Core 應用程式的原生 IIS 模組。</span><span class="sxs-lookup"><span data-stu-id="52b51-111">ANCM is a native IIS module that hooks into the IIS pipeline and redirects traffic to the backend ASP.NET Core application.</span></span> <span data-ttu-id="52b51-112">大部分的其他模組 (例如 Windows 驗證) 仍有機會執行。</span><span class="sxs-lookup"><span data-stu-id="52b51-112">Most other modules, such as windows authentication, still get a chance to run.</span></span> <span data-ttu-id="52b51-113">唯有當已針對要求選取處理常式，且處理常式對應定義於應用程式的 *web.config* 檔案中時，ANCM 才會取得控制權。</span><span class="sxs-lookup"><span data-stu-id="52b51-113">ANCM only takes control when a handler is selected for the request, and handler mapping is defined in the application *web.config* file.</span></span>

<span data-ttu-id="52b51-114">因為處理序中執行的 ASP.NET Core 應用程式會與 IIS 背景工作處理序分開，所以 ANCM 也會執行處理序管理。</span><span class="sxs-lookup"><span data-stu-id="52b51-114">Because ASP.NET Core applications run in a process separate from the IIS worker process, ANCM also does process management.</span></span> <span data-ttu-id="52b51-115">ANCM 會在第一個要求送入時啟動 ASP.NET Core 應用程式的處理序，並在其損毀時將它重新啟動。</span><span class="sxs-lookup"><span data-stu-id="52b51-115">ANCM starts the process for the ASP.NET Core application when the first request comes in and restarts it when it crashes.</span></span> <span data-ttu-id="52b51-116">此行為基本上與在 IIS 中執行同處理序，並由 WAS (Windows 啟用服務) 所管理的傳統 ASP.NET 應用程式相同。</span><span class="sxs-lookup"><span data-stu-id="52b51-116">This is essentially the same behavior as classic ASP.NET applications that run in-process in IIS and are managed by WAS (Windows Activation Service).</span></span>

<span data-ttu-id="52b51-119">要求自網路傳入並叫用核心模式 Http.Sys 驅動程式，該驅動程式會在主要連接埠 (80) 或 SSL 連接埠 (443) 上將要求路由至 IIS。</span><span class="sxs-lookup"><span data-stu-id="52b51-119">Requests come in from the Web and hit the kernel mode Http.Sys driver which routes them into IIS on the primary port (80) or SSL port (443).</span></span> <span data-ttu-id="52b51-120">ANCM 則會在針對應用程式設定的 HTTP 連接埠 (不是連接埠 80/443) 上，將要求轉送至 ASP.NET Core 應用程式。</span><span class="sxs-lookup"><span data-stu-id="52b51-120">ANCM forwards the requests to the ASP.NET Core application on the HTTP port configured for the application, which isn't port 80/443.</span></span>

<span data-ttu-id="52b51-121">Kestrel 會接聽來自 ANCM 的流量。</span><span class="sxs-lookup"><span data-stu-id="52b51-121">Kestrel listens for traffic coming from ANCM.</span></span>  <span data-ttu-id="52b51-122">ANCM 在啟動時透過環境變數指定連接埠，而 [UseIISIntegration](#call-useiisintegration) 方法則會設定伺服器來接聽 `http://localhost:{port}`。</span><span class="sxs-lookup"><span data-stu-id="52b51-122">ANCM specifies the port via environment variable at startup, and the [UseIISIntegration](#call-useiisintegration) method configures the server to listen on `http://localhost:{port}`.</span></span> <span data-ttu-id="52b51-123">有額外的檢查可拒絕不是來自 ANCM 的要求。</span><span class="sxs-lookup"><span data-stu-id="52b51-123">There are additional checks to reject requests not from ANCM.</span></span> <span data-ttu-id="52b51-124">(ANCM 不支援 HTTPS 轉送，因此即使由 IIS 透過 HTTPS 接收，要求還是會透過 HTTP 轉送。)</span><span class="sxs-lookup"><span data-stu-id="52b51-124">(ANCM doesn't support HTTPS forwarding, so requests are forwarded over HTTP even if received by IIS over HTTPS.)</span></span>

<span data-ttu-id="52b51-125">Kestrel 會收取來自 ANCM 的要求，並將它們推送到 ASP.NET Core 中介軟體管線，該管線接著會處理這些要求，並將它們作為 `HttpContext` 執行個體傳遞至應用程式邏輯。</span><span class="sxs-lookup"><span data-stu-id="52b51-125">Kestrel picks up requests from ANCM and pushes them into the ASP.NET Core middleware pipeline, which then handles them and passes them on as `HttpContext` instances to application logic.</span></span> <span data-ttu-id="52b51-126">然後，應用程式的回應會傳回 IIS，而 IIS 會將它們推送回起始要求的 HTTP 用戶端。</span><span class="sxs-lookup"><span data-stu-id="52b51-126">The application's responses are then passed back to IIS, which pushes them back out to the HTTP client that initiated the requests.</span></span>

<span data-ttu-id="52b51-127">ANCM 也有幾個其他函式：</span><span class="sxs-lookup"><span data-stu-id="52b51-127">ANCM has a few other functions as well:</span></span>

## <a name="proxy-configuration-uses-http-protocol-and-a-pairing-token"></a><span data-ttu-id="52b51-179">Proxy 組態使用 HTTP 通訊協定和配對權杖</span><span class="sxs-lookup"><span data-stu-id="52b51-179">Proxy configuration uses HTTP protocol and a pairing token</span></span>

<span data-ttu-id="52b51-180">ANCM 和 Kestrel 之間建立的 Proxy 會使用 HTTP 通訊協定。</span><span class="sxs-lookup"><span data-stu-id="52b51-180">The proxy created between the ANCM and Kestrel uses the HTTP protocol.</span></span> <span data-ttu-id="52b51-181">使用 HTTP 是一項效能最佳化作業，其中 ANCM 與 Kestrel 之間的流量會在網路介面外的回送位址進行。</span><span class="sxs-lookup"><span data-stu-id="52b51-181">Using HTTP is a performance optimization where the traffic between the ANCM and Kestrel takes place on a loopback address off of the network interface.</span></span> <span data-ttu-id="52b51-182">沒有從伺服器外的位置竊聽 ANCM 與 Kestrel 之間流量的風險。</span><span class="sxs-lookup"><span data-stu-id="52b51-182">There's no risk of eavesdropping the traffic between the ANCM and Kestrel from a location off of the server.</span></span>

<span data-ttu-id="52b51-183">配對權杖用來保證 Kestrel 所接收的要求已由 IIS 代理，而且不是來自其他來源。</span><span class="sxs-lookup"><span data-stu-id="52b51-183">A pairing token is used to guarantee that the requests received by Kestrel were proxied by IIS and didn't come from some other source.</span></span> <span data-ttu-id="52b51-184">ANCM 會建立配對權杖，並將其設定成環境變數 (`ASPNETCORE_TOKEN`)。</span><span class="sxs-lookup"><span data-stu-id="52b51-184">The pairing token is created and set into an environment variable (`ASPNETCORE_TOKEN`) by the ANCM.</span></span> <span data-ttu-id="52b51-185">配對權杖也會設定成每個代理要求的標頭 (`MSAspNetCoreToken`)。</span><span class="sxs-lookup"><span data-stu-id="52b51-185">The pairing token is also set into a header (`MSAspNetCoreToken`) on every proxied request.</span></span> <span data-ttu-id="52b51-186">IIS 中介軟體會檢查其收到的每個要求，以確認配對權杖的標頭值符合環境變數值。</span><span class="sxs-lookup"><span data-stu-id="52b51-186">IIS Middleware checks each request it receives to confirm that the pairing token header value matches the environment variable value.</span></span> <span data-ttu-id="52b51-187">如果權杖值不相符，將記錄並拒絕要求。</span><span class="sxs-lookup"><span data-stu-id="52b51-187">If the token values are mismatched, the request is logged and rejected.</span></span> <span data-ttu-id="52b51-188">配對權杖的環境變數和 ANCM 與 Kestrel 之間的流量無法從伺服器外的位置進行存取。</span><span class="sxs-lookup"><span data-stu-id="52b51-188">The pairing token environment variable and the traffic between the ANCM and Kestrel aren't accessible from a location off of the server.</span></span> <span data-ttu-id="52b51-189">在不知道配對權杖值的情況下，攻擊者無法略過 IIS 中介軟體的檢查送出要求。</span><span class="sxs-lookup"><span data-stu-id="52b51-189">Without knowing the pairing token value, an attacker can't submit requests that bypass the check in the IIS Middleware.</span></span>
